# Customers API Module Documentation

## Overview

This module provides production-ready CRUD APIs for managing customers.

It includes:

- Create Customer
- List Customers (Dynamic Filtering + Pagination + Role Scope)
- Get Customer by ID
- Edit Customer (Dynamic Update)

The system follows:

- Schema-level validation (Pydantic)
- Role-based access control (RBAC)
- Database-level integrity enforcement
- Structured logging (request_id + emp_id)
- Secure dynamic query construction
- Strict response validation

---

# Architecture Principles

## 1. Validation Responsibility Split

### Schema-Level Validation (Pydantic)

Handled in `CustomerIn` and `CustomerEditIn`:

- Field length checks
- Email validation
- Mobile regex validation
- URL validation
- XSS sanitation
- Required fields logic

### API-Level Validation

Handled inside route functions:

- Date sanity checks
- Empty update checks
- Business logic enforcement

### Database-Level Validation

Handled by PostgreSQL:

- UNIQUE constraints (email, mobile)
- FOREIGN KEY constraints (rm_id, op_id)
- NOT NULL constraints
- Default values (is_active)

---

# Logging Strategy

Each request includes:

- `request_id` (UUID)
- `emp_id` (from JWT)

Structured logger:

```python
log = logging.LoggerAdapter(
    logger,
    {"request_id": request_id, "emp_id": emp_id},
)
```

This enables:

- Traceability
- Audit tracking
- Production debugging
- Per-user monitoring

# API Endpoints

---

## 1. Create Customer

### Endpoint

```
POST /api/v1/customers
```

### Access

Requires:

```
EMPLOYEE WRITE
```

### Features

- Schema validation before execution
- Masked logging for sensitive data
- Database handles uniqueness
- Transaction-safe insert
- Foreign key validation
- Strict response validation
- Proper HTTP error mapping

### Possible Responses

| Status | Meaning |
|--------|----------|
| 201 | Customer created |
| 409 | Duplicate email/mobile |
| 400 | Invalid rm_id/op_id |
| 500 | Database error |

---

## 2. List Customers (Dynamic Filtering)

### Endpoint

```
GET /api/v1/customers/customers_dyn_filter
```

### Access

Requires:

```
EMPLOYEE READ
```

### Features

- Dynamic filtering
- Role-based scope restriction
- Pagination
- Date filtering
- Active/inactive filtering
- Parameterized SQL (no injection risk)
- Strict response validation

---

### Business Filters (User Controlled)

- customer_id
- full_name (ILIKE)
- email (ILIKE)
- mobile
- business_type
- state
- city
- rm_id (admin only)
- op_id (admin only)
- is_active
- from_date
- to_date
- limit
- offset

---

### Role-Based Scope Filtering (System Controlled)

| Role | Visibility |
|------|------------|
| ADMIN | All customers |
| RM | Only customers where rm_id = emp_id |
| OP | Only customers where op_id = emp_id |
| Both roles | Self-scoped via OR condition |

Security filtering is always applied and cannot be bypassed by query parameters.

---

### Response

Returns list of:

```
CustomerOut
```

With:

```
message = "Customers listed successfully."
```

## 4. Edit Customer (Dynamic Update)

### Endpoint

```
POST /api/v1/customers/{customer_id}/customer_dyn/edit
```

### Access

Requires:

```
EMPLOYEE WRITE
```

### Features

- Dynamic update (only provided fields)
- Uses `exclude_unset=True`
- Transaction safe
- DB-level uniqueness enforcement
- DB-level foreign key enforcement
- Automatic updated_at update
- Strict response validation
- Structured logging

---

### Update Logic

```python
update_data = payload.model_dump(exclude_unset=True)
```

Only provided fields are updated.

If no fields provided → 400 error.

---

### DB-Level Protections

- UNIQUE(email, mobile)
- FOREIGN KEY(rm_id, op_id)
- NOT NULL
- Default values

# Security Model

The system enforces two types of filtering:

## 1. Business Filtering

User-controlled filters such as:

- city
- state
- mobile
- date range

## 2. Security Scope Filtering

System-controlled filters based on role:

- rm_id restriction
- op_id restriction
- admin bypass

Final query:

```
Business Filters
+ Security Scope
+ Status Filter
+ Date Filter
+ Pagination
```

Security filtering is never optional.
